# /incident-report — 서비스 장애 보고서 자동 생성 및 사후 분석

서비스 다운, API 오류, 데이터 손실 등 장애 발생 시 타임라인 기록, 영향 범위 분석, 근본 원인 분석(RCA), 재발 방지 계획을 포함한 표준 장애 보고서를 생성한다.

## 실행 단계

1. **장애 기본 정보 수집**
   - `$ARGUMENTS`에서 장애 유형, 발생 시간, 감지 방법 파싱
   - 자동 수집 (가능한 경우):
     - Vercel 배포 로그
     - Sentry 에러 로그: `C:/oomni-data/logs/errors/`
     - Uptime 모니터링 알림 기록

2. **장애 분류**
   심각도(Severity) 분류:
   - **SEV-1 (Critical)**: 전체 서비스 다운, 데이터 손실
   - **SEV-2 (High)**: 주요 기능 불능, 결제 실패
   - **SEV-3 (Medium)**: 부분 기능 저하, 성능 저하
   - **SEV-4 (Low)**: 마이너 버그, UI 이슈

3. **타임라인 재구성**
   ```
   HH:MM:SS - [감지] 모니터링 알림 수신
   HH:MM:SS - [확인] 장애 확인 및 SEV 분류
   HH:MM:SS - [조치] 첫 번째 대응 조치
   HH:MM:SS - [진행] ...
   HH:MM:SS - [해결] 서비스 정상화 확인
   ```
   총 장애 시간(TTR: Time To Resolve) 계산

4. **영향 범위 분석**
   - 영향받은 사용자 수 (로그인 실패, 에러 발생 사용자)
   - 영향받은 기능 목록
   - 영향받은 기간
   - 예상 수익 손실 (ARPU × 영향 사용자 수 × 장애 시간/720시간)
   - 고객 문의 수 (이메일, 채팅)

5. **근본 원인 분석 (RCA — 5 Whys)**
   ```
   Why 1: 왜 서비스가 다운됐는가? → DB 연결 실패
   Why 2: 왜 DB 연결이 실패했는가? → 커넥션 풀 고갈
   Why 3: 왜 커넥션 풀이 고갈됐는가? → N+1 쿼리 문제
   Why 4: 왜 N+1 쿼리가 생겼는가? → 코드 리뷰 미흡
   Why 5: 왜 코드 리뷰가 미흡했는가? → 테스트 커버리지 부족
   근본 원인: 테스트 커버리지 부족으로 인한 성능 이슈 미감지
   ```

6. **해결 과정 기록**
   - 시도한 해결 방법 (실패한 것 포함)
   - 최종 적용한 해결 방법
   - 적용 코드/설정 변경사항

7. **재발 방지 계획**
   단기/중기/장기 조치:
   - **즉시 (당일)**: 임시 방편 조치
   - **단기 (1주일)**: 근본 원인 코드 수정
   - **중기 (1개월)**: 모니터링/알림 강화
   - **장기 (3개월)**: 아키텍처 개선

8. **고객 공지 초안 생성**
   - 서비스 페이지 공지 문구
   - 이메일 사과 메시지
   - 트위터/SNS 공지

9. **보고서 저장 및 팀 공유**
   - 내부 보고서 저장
   - Slack/Telegram으로 요약 발송

## 출력 형식

### 장애 보고서 (`incident-[ID]_YYYY-MM-DD.md`)

```markdown
# 장애 보고서 INC-2026-042

**심각도**: SEV-2 (High)
**상태**: 해결됨 ✅
**발생 시간**: 2026-04-05 14:23:00 KST
**해결 시간**: 2026-04-05 15:47:00 KST
**총 장애 시간**: **84분**

---

## 요약

Stripe 웹훅 처리 실패로 인해 결제 완료 후 구독 활성화가 되지 않는 문제 발생.
84분 동안 17명의 신규 결제 고객이 서비스를 이용하지 못함.

---

## 타임라인

| 시간 | 이벤트 | 담당 |
|------|--------|------|
| 14:23 | Sentry에서 webhook 처리 오류 알림 수신 | 자동 |
| 14:25 | 장애 확인 및 SEV-2 분류 | 장우경 |
| 14:30 | Stripe 대시보드에서 웹훅 실패 로그 확인 | 장우경 |
| 14:45 | 임시 조치: 수동으로 구독 활성화 스크립트 실행 | 장우경 |
| 15:10 | 근본 원인 파악: STRIPE_WEBHOOK_SECRET 환경변수 누락 | 장우경 |
| 15:15 | Vercel에 환경변수 추가 후 재배포 | 장우경 |
| 15:47 | 서비스 정상화 확인 | 장우경 |

**감지 시간 (MTTD)**: 2분
**해결 시간 (MTTR)**: 84분

---

## 영향 범위

- **영향받은 사용자**: 17명 (결제 완료 후 서비스 이용 불가)
- **영향받은 기능**: 신규 구독 활성화
- **예상 수익 손실**: ₩0 (결제는 완료됨, 서비스 지연만 발생)
- **고객 문의**: 3건 (이메일 2건, 채팅 1건)

---

## 근본 원인 분석 (5 Whys)

1. 왜 구독 활성화가 안 됐는가? → Stripe 웹훅 서명 검증 실패
2. 왜 서명 검증이 실패했는가? → 환경변수 `STRIPE_WEBHOOK_SECRET` 없음
3. 왜 환경변수가 없었는가? → 새 Vercel 프로젝트 생성 시 누락
4. 왜 누락됐는가? → 환경변수 체크리스트가 없음
5. 왜 체크리스트가 없는가? → 배포 프로세스가 문서화되지 않음

**근본 원인**: 배포 체크리스트 부재

---

## 해결 방법

1. Vercel 대시보드에서 `STRIPE_WEBHOOK_SECRET` 환경변수 추가
2. Vercel 재배포 (`vercel --prod`)
3. 영향받은 17명의 구독 수동 활성화 스크립트 실행:
   ```bash
   npx tsx scripts/fix-pending-subscriptions.ts --date 2026-04-05
   ```

|------|------|------|------|
| 환경변수 체크리스트 문서화 | D+1 | 장우경 | ✅ |
| 웹훅 엔드포인트 헬스체크 모니터링 추가 | D+3 | 장우경 | ⏳ |
| 배포 전 환경변수 검증 스크립트 | D+7 | 장우경 | 📋 |
| 결제 파이프라인 E2E 테스트 작성 | D+14 | 장우경 | 📋 |

```

## 저장 위치

- `C:/oomni-data/ops/incidents/incident-[ID]_YYYY-MM-DD.md`
- `C:/oomni-data/ops/incidents/incident-[ID]_YYYY-MM-DD.json`
- `C:/oomni-data/ops/incidents/index.json` (전체 장애 목록)

## 추가 인자

`$ARGUMENTS` — 다음 형식으로 입력:
- `[장애 설명]`
- `--severity sev1|sev2|sev3|sev4` : 심각도
- `--started "2026-04-05 14:23"` : 시작 시간
- `--resolved "2026-04-05 15:47"` : 해결 시간
- `--affected-users 17` : 영향받은 사용자 수
- `--notify-customers` : 고객 공지 이메일 초안 생성
- 예시: `/incident-report "Stripe 웹훅 처리 실패로 구독 활성화 불가" --severity sev2 --started "2026-04-05 14:23" --resolved "2026-04-05 15:47" --notify-customers`
