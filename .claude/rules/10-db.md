# 10 — DB / SQLite 규칙

## v3.0.0 DB 전략 (핵심 3원칙)

1. **migration 히스토리 완전 제거** → 단일 `SCHEMA_SQL`만 존재 (v1~v9 전부 삭제)
2. **ALTER TABLE RENAME 절대 금지** → FK 오염 근본 원인
3. **migration은 ADD COLUMN만** → DROP/RENAME은 major 버전업 시 DB 리셋으로만 처리

## v2.x DB 감지 및 처리

- 기존 v2.x DB 감지 시 → Electron dialog: "v3.0 업그레이드 필요, 기존 DB 백업 후 리셋"
- 자동 백업 경로: `C:/oomni-data/oomni-backup-{timestamp}.db`
- 리셋 후 `SCHEMA_SQL` 단일 실행으로 초기화

- ❌ `ALTER TABLE foo RENAME TO foo_bak` (PRAGMA legacy_alter_table 없이)
- ❌ 신규 테이블 migration chain으로 추가 (SCHEMA_SQL에 직접 추가)
- ❌ subscriptions / payment_logs 테이블 (Phase 2)
- ❌ `postMigrationFkRepair(db: Database)` 패턴 — 파라미터 섀도잉 버그 (v2.9.21 근본 원인)
